## Failure taxonomy (the binding contract)

`apps/api/src/modules/billing/services/billing-failure.taxonomy.ts` is the single
source of truth. Every catch block / charge-status branch routes through it.

| Class | Trigger | `shouldDun` | Effect |
| --- | --- | --- | --- |
| A `TRANSPORT` | thrown error (network/timeout/5xx/config); unclassified errors fail-safe here | ❌ | invoice stays `ISSUED`; retry next sweep |
| B `DECLINE` | returned `ChargeResult.status === "FAILED"` / `PAYMENT_FAILED` callback | ✅ | invoice `FAILED`; dunning advances |
| C `ABANDON` | hosted redirect not collectable via rebill | ❌ | leave open |
| D `PENDING` | `PENDING_CONFIRMATION` (3DS / async) | ❌ | leave `ISSUED`; await callback |
| E `DUPLICATE` | Prisma P2002 / replayed provider tx | ❌ | idempotent no-op |
| — `MANUAL_RENEWAL` | `UnsupportedOperationError` (CMI / offline) | ❌ | send manual-pay notice |

**Regression to keep green:** a FREE merchant hitting a Payzone transport failure
must never show `PAST_DUE` (class A, not B).

## Edge-case coverage notes (master catalog deltas)

- **Period math** anchored to `Africa/Casablanca` for cron triggers; all stored timestamps remain UTC. Month/year rollover uses `setUTCMonth`/`setUTCFullYear` (handles 31st → short months by JS normalisation; verify Feb behaviour in tests).
- **Zero-amount / FREE** renewals keep the existing zero-invoice + period-advance path (no provider call).
- **ENTERPRISE annual** still throws "custom quote" in `resolvePrice` — null/manual pricing path unchanged.
- **Invoice lead override** `<= 0` or non-integer falls back to the safe default (pre-renewal invoicing can never be silently disabled).

### Slice B (OPS-4) reconciliation + callback security deltas

- **Scheduler mechanism (decision):** reconciliation runs as a daily `@Cron` (`BILLING_RECONCILIATION_DAILY_CRON = "30 6 * * *"`, Casablanca), 30 min after the renewal sweep so a just-`PENDING_CONFIRMATION` rebill has settled at the acquirer before we consult it. Chosen over a `q.billing` BullMQ worker to stay consistent with `BillingRenewalCron`; the BullMQ migration (with cross-instance Redis dedup) remains tracked as F-3/F-11. Reconciliation is idempotent, so a multi-replica double run is a no-op.
- **Which invoices reconcile (refinement of F-4's wording):** only `ISSUED` `RENEWAL` invoices **whose `providerTxId` is set** are consulted — i.e. a charge was actually attempted (sync rebill returned PENDING, or a callback was lost). A pre-issued lead-time invoice with no charge yet has `providerTxId = null` and is left to the renewal **charge** sweep; reconciliation never charges, only consults. This prevents settling a fresh invoice against the previous cycle's anchor transaction.
- **PZ-6 (do not advance on PENDING):** a consulted `PENDING` / `UNKNOWN` / `CANCELLED` / `REFUNDED` outcome dispatches **nothing** and leaves the invoice `ISSUED` for the next sweep. Only `SUCCESS → PAYMENT_SUCCESS` and a hard `FAILED → PAYMENT_FAILED` are dispatched — `CANCELLED`/abandon is class C and must never dun (taxonomy-faithful).
- **Amount guard charge basis (see F-10):** the guard accepts an event amount equal to the invoice `subtotalCents` *or* `totalCents`. Today's rebill collects `subtotalCents` (`priceCents`, no VAT), so a strict `== totalCents` check would reject every legitimate Payzone settlement. An event with **no** amount cannot be tamper-checked and is allowed through (unchanged pre-5.8.2 behaviour).
- **Rejection posture:** an integrity-gate rejection on the **callback** path propagates `BillingSecurityError` out of the dispatcher → the controller returns non-2xx (fail-closed; the provider may retry but a genuinely tampered event can never settle). On the **reconciliation** path the per-invoice loop logs + isolates the throw and continues the sweep. Both record a CRITICAL `SecurityAuditPort` entry when the port is bound.

### Slice C (OPS-5) comms + PDF-off-hot-path deltas

- **Cancellation effective-date copy (the point of two kinds):** `SUBSCRIPTION_CANCEL_SCHEDULED` carries the **future** period-end date ("access until …, then free plan") so a merchant who cancels mid-cycle is not alarmed into thinking access stops today; `SUBSCRIPTION_CANCELLED` carries **today's** date and confirms the transition. The scheduled→cancelled pair therefore fires twice across a cancel-at-period-end lifecycle (once on request, once when the boundary downgrade lands) — this is intentional, not a duplicate.
- **`planName` is the pre-downgrade tier.** Both mails render `subscription.tier` captured **before** the FREE downgrade write, so the merchant sees the plan they actually cancelled ("STARTER"), not "FREE".
- **Offline instructions reuse the adapter seam, not new adapter code.** Only the `OFFLINE_PAYMENT_NOTIFIER` binding changed (log bridge → `OfflineInstructionMailer`); `AbstractOfflineAdapter.safeDispatch` still swallows a transient mail failure so the persisted `OfflinePaymentRequest` is never rolled back by an email hiccup — instructions can be resent from the console. The new offline mail kinds carry `customerName` in the payload (greeting), unlike the renewal mails.
- **PDF receipt timing (documented choice):** the receipt email is enqueued by the **worker after** the PDF is uploaded and its URL attached to the invoice row — so the `INVOICE` mail always links a live PDF. We did **not** send an immediate "receipt without PDF" and back-fill; a merchant gets exactly one receipt, when the artifact exists. Render+receipt now also fires on the **sync-rebill** success path (`markRenewalSuccess`), which previously sent nothing.
- **Idempotency unchanged by the async move.** The handler still resolves the invoice by `providerTxId` and enqueues `{ invoiceId }`; a duplicate callback (PZ-3/PZ-7) is still neutralised upstream in `DunningService.processPaymentSuccess` (the already-PAID short-circuit) before any job is enqueued, so a replayed callback cannot enqueue a second render/receipt. The `q.invoice` worker is itself safe to re-run (render+upload is overwrite-idempotent on the same storage key).
- **Worker DLQ:** an invoice-PDF job that exhausts `QUEUE_TUNING.invoice.attempts` dead-letters to `q.invoice.dlq` with the failure reason, mirroring the mail worker; a permanently failing PDF therefore never blocks the queue and is recoverable for ops.

### Slice D (OPS-6) PWA self-service deltas

- **`payOpenInvoice` lives in `DunningService`, not a new service.** Settling a paid invoice must reuse the *exact* dunning write paths (`markRenewalSuccess` → PAID + period advance + receipt job; `recordPaymentFailure` → dun) so there is never a second activation code path to keep in sync. The controller stays thin; `SubscriptionService` keeps the two read/anchor methods (`getBillingSummary`, `refreshPaymentMethod`).
- **Charge basis = `subtotalCents` (consistent with the renewal sweep).** Pay-open-invoice rebills/charges the subtotal (`priceCents`), matching `processRenewal` and the F-9 amount guard (subtotal *or* total accepted). It deliberately does **not** pre-empt the F-10 VAT-collection decision — when F-10 lands, both the sweep and pay-open-invoice change together.
- **Summary status is pre-normalised server-side.** `getBillingSummary` collapses `FREE + PAST_DUE → TRIALING` and `DELINQUENT → SUSPENDED` exactly like the session's `deriveBillingState`, so the PWA never has to special-case the free plan to avoid a "paiement en retard" banner — the rule is enforced once, on the server.
- **Card refresh ships with explicit charge disclosure (F-12 closed).** Payzone requires a captured sale for a rebill anchor — not ccAuth-only. Update card performs a real Payment-Page charge at `priceCents`, records a PAID `PAYMENT_METHOD_UPDATE` invoice, and never advances the period (`CARD_UPDATE` intent guard). PWA routes refresh through a confirm dialog stating the charge amount and that the billing period is unchanged. Disable via `BILLING_CARD_UPDATE_ENABLED=false`. Optional charge-free path → **F-12b**.
- **CMI-2 enforced at pay time.** An `ISSUED` invoice bound to a provider other than the subscription's active gateway is rejected (`409 PROVIDER_MISMATCH`) — a merchant cannot accidentally settle a CMI invoice through Payzone.
- **Return page is UX catch-up, not source of truth.** `/billing/success` reloads the session + invalidates the billing queries, but the authoritative activation still comes from the verified Payzone callback (Slice A/B). A merchant who closes the tab before returning is still reconciled by the daily job (C-9).

### Slice E (OPS-7 / OPS-8) escalation + refund deltas

- **Two grace windows, deliberately (F-7 / C-15).** A manual-renewal merchant gets a **manual-pay** window (`periodStart + GRACE_DAYS`) during which the `ISSUED` invoice simply sits open — no dunning, because they may still pay. Only once that elapses does `recordPaymentFailure` open the **standard** dunning grace and walk PAST_DUE → DELINQUENT. We did **not** collapse the two into one window: the dunning table is the binding contract and an offline/CMI merchant who hasn't paid yet must not be locked the instant the period lapses. Net effect: an *ignored* CMI/offline renewal reaches DELINQUENT later than a Payzone decline — acceptable, since there is no card to decline.
- **Escalation reuses `recordPaymentFailure`, never a parallel path.** The same write path that duns a Payzone decline duns an ignored manual renewal — grace anchoring (C-7), retry cap, `PAYMENT_FAILED` mail, and the read-only lock all stay single-sourced. The early grace-expiry guard in `processRenewal` (PAST_DUE + elapsed grace → DELINQUENT) continues to handle subsequent sweeps for all providers.
- **Offline request creation is idempotent on "one awaiting per subscription".** `ensureOfflineRenewalRequest` creates the `AWAITING_CONFIRMATION` request only when none is already awaiting, so a daily sweep never mints duplicates. CMI gets the emailed pay link only (its `rebill` throws by contract — never called). A rebill failure (e.g. a missing anchor) is logged, never dunned.
- **Admin confirm settles the *existing* renewal invoice, not a new one.** The confirm controller resolves the open `ISSUED` `RENEWAL` invoice and dispatches the synthetic `PAYMENT_SUCCESS` against **its** id, so the standard `PaymentSuccessHandler` marks it PAID + advances the period (and the amount-tamper guard passes because the offline request's `amountCents` equals the invoice subtotal). Initial-subscribe confirm has no such invoice and resolves by subscription id, minting the first CYCLE invoice exactly as before — no behaviour change for new subscriptions.
- **Refund routing is provider-faithful (C-16).** Payzone reverses at the acquirer (`adapter.refund`); CMI records a credit note + flags the manual back-office step (`status: PENDING`, adapter never called because it throws); offline acknowledges + credit notes. The **original** invoice's `meta` is the refund ledger of record (PZ-11) — partial refunds accumulate and can never exceed `totalCents`. The credit-note invoice's `totalCents` clamps to 0 by the shared money guard (tracked as **F-13**); it is an internal audit artefact, never an open/charged invoice.
- **Refunds are admin-only and never automatic.** A merchant cannot self-refund (no merchant-audience endpoint). Mid-cycle cancel still does **not** refund — a proration review is a separate, deliberate admin refund call, per the spec's "never auto-refund on cancel without proration review".
